Superuser and Permission Management in Linux

Learn about Linux user account types including superuser, normal user, and system user, along with privilege escalation using su and sudo commands and sudoers configuration.

November 21, 2021 Updated July 16, 2026 6 min read
Linux Commands

In Linux systems, different permissions are assigned to user accounts to maintain security and stability.

Types of User Accounts

  • Superuser:
    • A user account typically called root.
    • Has full administrative privileges over the system. Can perform any operation including creating, modifying, and deleting files, managing users, and changing system settings.
    • Since it can perform operations that affect the core of the system, incorrect operations may impact the entire system. It is best practice to avoid logging in directly as root and instead temporarily elevate privileges only when necessary.
  • Normal User:
    • A user account for performing everyday tasks.
    • Has limited privileges on the system, and can only operate within their own home directory and other permitted areas.
    • From a security perspective, it is recommended to use a normal user account for daily operations.
  • System User:
    • A user account automatically created to run specific services and applications (e.g., apache, mysql, daemon).
    • Not intended for direct human login; only the minimum privileges required by the service or application are granted.

Privilege Escalation Commands

These commands allow a normal user to temporarily borrow superuser privileges to execute commands.

  • su (substitute user / switch user):
    • A command to switch from the current user to another user (default is root).
    • Requires the password of the target user.
    su - [username] # The hyphen loads the target user's environment variables
# Example: su - root (switch to the root user)
# Example: su - user_name (switch to user_name)
  • sudo (superuser do):
    • A command that allows executing commands that require superuser privileges temporarily.
    • Requires the password of the user executing sudo.
    • Users who can use sudo are managed in the /etc/sudoers file.
    sudo [options] [command]
# Example: sudo apt update (update package list)
# Example: sudo systemctl restart apache2 (restart Apache service)

sudo Configuration (/etc/sudoers)

The users who can execute the sudo command and the commands they can run are defined in the /etc/sudoers file. This file is critical, and syntax errors can cause system problems. Therefore, always use the visudo command to edit it instead of editing it directly.

sudo visudo

visudo is a command for safely editing the /etc/sudoers file. It checks for syntax errors during editing and saves the file only if there are no issues.

Common configuration examples:

# The root user can execute all commands without a password
root    ALL=(ALL:ALL) ALL

# Users in the wheel group can execute all commands without a password
%wheel  ALL=(ALL:ALL) NOPASSWD: ALL

# A specific user can execute specific commands without a password
your_username ALL=(ALL) NOPASSWD: /usr/bin/apt update, /usr/bin/apt upgrade

Specifying NOPASSWD: allows skipping password entry when executing those commands. It is important to configure this with minimal scope, considering security risks.

Where sudo executions get logged

sudo always records who ran what as root, and when. This isn’t a minor extra feature — it’s one of the main operational reasons sudo is preferred over su (once you switch to root via su, individual subsequent actions become much harder to attribute to the original user).

On Debian/Ubuntu, sudo invocations are logged via syslog to /var/log/auth.log; on RHEL/CentOS-family systems, to /var/log/secure. A typical line looks like this:

Jul 16 09:12:31 web01 sudo:   yuhi-sa : TTY=pts/0 ; PWD=/home/yuhi-sa ; USER=root ; COMMAND=/usr/bin/apt update

Note: The line above is an illustrative example showing the general format of a Debian-family sudo log entry. It was not captured live from this article’s verification environment (a sandbox running macOS). To inspect real logs on a target server, filter as follows:

# Extract only recent sudo invocations
grep sudo /var/log/auth.log | tail -n 20

# On systemd-based distributions, journalctl works too
journalctl _COMM=sudo --since "1 hour ago"

For stricter auditing, auditd can record the actual command execution (execve) at the system-call level.

# Install and enable auditd (Debian/Ubuntu example)
sudo apt install auditd
sudo systemctl enable --now auditd

# Add a rule to watch execve calls made as root (uid 0), including via sudo
sudo auditctl -a always,exit -F arch=b64 -S execve -F euid=0 -F auid!=-1 -k sudo_exec

# Search the recorded events
sudo ausearch -k sudo_exec | tail -n 30

Because auditd also records auid (the Audit UID, fixed at login time), it can trace back to “who was actually logged in” even after they temporarily became root via sudo.

Note that this article was written in a sandboxed environment (macOS, where password-based sudo cannot be executed), so the log line and commands above are illustrative examples of typical Debian/Ubuntu behavior rather than output actually captured while running as root.

Benchmark: measuring the per-invocation overhead of a “wrapper command”

sudo doesn’t just execute a command directly — it interposes password authentication, the PAM stack, logging, and environment-variable sanitization before launching the target command. In other words, sudo command is not “running the bare command,” but “running the command through an extra processing layer.”

Because this environment has no root password configured, sudo itself could not actually be invoked, so the overhead of sudo specifically could not be measured here. Instead, this article measures the general principle — that inserting a thin wrapper layer around command execution adds a measurable per-invocation cost — using something that actually could be run in this environment.

Three ways of launching /usr/bin/true were compared:

  • Direct exec: subprocess.run(["/usr/bin/true"]) — the Python process directly forks and execs
  • Shell wrapper (shell=True): subprocess.run("/usr/bin/true", shell=True) — an extra sh -c process sits in between
  • Shell wrapper (os.system): os.system("/usr/bin/true") — this also goes through sh -c internally
import subprocess
import time
import os
import statistics

N = 500

def bench(fn):
    for _ in range(10):
        fn()  # warm-up
    times = []
    for _ in range(N):
        t0 = time.perf_counter()
        fn()
        t1 = time.perf_counter()
        times.append((t1 - t0) * 1000)  # ms
    return statistics.mean(times), statistics.stdev(times)

direct = bench(lambda: subprocess.run(["/usr/bin/true"], check=True))
shell = bench(lambda: subprocess.run("/usr/bin/true", shell=True, check=True))
system = bench(lambda: os.system("/usr/bin/true"))

Running each method 500 times on macOS (arm64) in this sandbox (measured twice to confirm reproducibility) gave the following results:

Invocation methodMean timeStd. dev.
Direct exec (subprocess.run)1.79 ms0.11 ms
Shell wrapper (shell=True)5.44 ms0.20 ms
Shell wrapper (os.system)4.81 ms0.30 ms

Just adding a single sh -c layer around the direct exec increased the mean invocation time by roughly 2.7–3.0x. This comes from the additional fork/exec of the shell process itself plus shell initialization cost.

sudo’s internal processing (initializing the PAM authentication stack, parsing /etc/sudoers, writing to syslog, clearing and setting up environment variables) is heavier than the single sh -c layer measured here, so sudo command in a real deployment is noticeably slower than a bare direct execution. The exact magnitude depends on things like whether the authentication timestamp cache is warm (sudo caches successful authentication for a few minutes by default) and how PAM is configured, so an accurate multiplier would need to be measured on the target system. What’s shown here is a real measurement of the general principle — that a wrapper layer has a measurable cost — not a sudo-specific number.
