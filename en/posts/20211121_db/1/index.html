SQL Basics: The Standard Language for Database Operations

An overview of SQL command categories: DML (SELECT, INSERT, UPDATE, DELETE), DDL (CREATE, ALTER, DROP), and DCL (GRANT, REVOKE, COMMIT), with runnable Python/sqlite3 demos and a measured CREATE INDEX benchmark at 1M rows.

November 21, 2021 Updated July 16, 2026 5 min read
SQL Database

SQL (Structured Query Language) is the standard language for operating relational databases. SQL commands are broadly classified into three categories.

1. Data Manipulation Language (DML)

Commands for manipulating data within a database.

  • SELECT: Retrieves records (rows) from a database. Results can also be joined.
  • INSERT: Inserts new records into a database table.
  • UPDATE: Updates existing records in a database.
  • DELETE: Deletes records from a database.

2. Data Definition Language (DDL)

Commands for defining, modifying, and deleting database structures and objects (tables, views, indexes, etc.).

  • CREATE: Creates new database objects (e.g., CREATE TABLE, CREATE VIEW, CREATE INDEX).
  • ALTER: Modifies the structure of existing database objects (e.g., ALTER TABLE to add, remove, or modify columns).
  • DROP: Deletes existing database objects (e.g., DROP TABLE, DROP VIEW).
  • RENAME: Renames database objects.
  • TRUNCATE: Deletes all records from a table while keeping the table structure. Unlike DELETE, it usually cannot be rolled back and is faster.

3. Data Control Language (DCL)

Commands for managing access privileges to the database and controlling transactions.

  • COMMIT: Confirms changes made in the current transaction and permanently saves them to the database.
  • ROLLBACK: Undoes changes made in the current transaction, reverting to the state at the start of the transaction.
  • SAVEPOINT: Sets a temporary save point within a transaction. You can ROLLBACK to this save point.
  • GRANT: Grants access privileges to database objects for users or roles.
  • REVOKE: Revokes access privileges to database objects from users or roles.

By combining these commands, you can perform various operations including database design, data management, and security configuration.

4. Running Real SQL to Check It Out

Abstract descriptions only get you so far, so let’s actually run some code. You don’t need to install MySQL or PostgreSQL to try the same SQL syntax — Python’s built-in sqlite3 module lets you run it right away (the SQL in this article follows common dialects like MySQL, but it’s demonstrated here with SQLite for reproducibility).

import sqlite3

conn = sqlite3.connect(":memory:")
cur = conn.cursor()

# DDL: CREATE
cur.execute("""
    CREATE TABLE employees (
        id INTEGER PRIMARY KEY,
        name TEXT NOT NULL,
        department TEXT,
        salary INTEGER
    )
""")

# DML: INSERT
cur.executemany(
    "INSERT INTO employees (name, department, salary) VALUES (?, ?, ?)",
    [
        ("Alice", "Engineering", 720000),
        ("Bob", "Sales", 580000),
        ("Carol", "Engineering", 810000),
        ("Dave", "Marketing", 540000),
    ],
)
conn.commit()

# DML: SELECT
cur.execute("SELECT id, name, salary FROM employees WHERE department = 'Engineering'")
print(cur.fetchall())

The actual output:

[(1, 'Alice', 720000), (3, 'Carol', 810000)]

Running UPDATE and DELETE next shows records actually being rewritten and removed.

# DML: UPDATE
cur.execute("UPDATE employees SET salary = salary + 30000 WHERE name = 'Bob'")
conn.commit()
cur.execute("SELECT name, salary FROM employees WHERE name = 'Bob'")
print(cur.fetchall())  # -> [('Bob', 610000)]

# DML: DELETE
cur.execute("DELETE FROM employees WHERE name = 'Dave'")
conn.commit()
cur.execute("SELECT name FROM employees ORDER BY id")
print(cur.fetchall())  # -> [('Alice',), ('Bob',), ('Carol',)]

Adding a column with ALTER TABLE changes the table structure itself, which you can confirm with PRAGMA table_info.

cur.execute("ALTER TABLE employees ADD COLUMN hire_year INTEGER")
conn.commit()
cur.execute("PRAGMA table_info(employees)")
for row in cur.fetchall():
    print(row)

(0, 'id', 'INTEGER', 0, None, 1)
(1, 'name', 'TEXT', 1, None, 0)
(2, 'department', 'TEXT', 0, None, 0)
(3, 'salary', 'INTEGER', 0, None, 0)
(4, 'hire_year', 'INTEGER', 0, None, 0)

All of this is the exact output from actually running the code above. Running each DML/DDL command makes it obvious at a glance whether it’s changing “data” or “structure.”

5. Measuring the Effect of an Index

How much does the CREATE INDEX command from the DDL section actually speed up queries? I built tables of 100,000 and 1,000,000 rows, ran the same WHERE query with and without an index, and averaged 30 runs with time.perf_counter().

import sqlite3
import time
import random
import statistics

random.seed(42)
conn = sqlite3.connect(":memory:")
cur = conn.cursor()
cur.execute("""
    CREATE TABLE users (
        id INTEGER PRIMARY KEY,
        email TEXT,
        age INTEGER,
        country TEXT
    )
""")
countries = ["JP", "US", "CN", "IN", "BR", "DE", "FR", "GB", "KR", "CA"]
rows = (
    (i, f"user{i}@example.com", random.randint(18, 80), random.choice(countries))
    for i in range(1_000_000)
)
cur.executemany("INSERT INTO users VALUES (?, ?, ?, ?)", rows)
conn.commit()

query = "SELECT id, email FROM users WHERE age = 42"

# EXPLAIN QUERY PLAN before an index exists
cur.execute("EXPLAIN QUERY PLAN " + query)
print(cur.fetchall())

times = []
for _ in range(30):
    start = time.perf_counter()
    cur.execute(query)
    cur.fetchall()
    times.append(time.perf_counter() - start)
print(f"before index: {statistics.mean(times) * 1000:.2f} ms")

# Create the index
cur.execute("CREATE INDEX idx_users_age ON users(age)")
conn.commit()

# EXPLAIN QUERY PLAN after the index exists
cur.execute("EXPLAIN QUERY PLAN " + query)
print(cur.fetchall())

times = []
for _ in range(30):
    start = time.perf_counter()
    cur.execute(query)
    cur.fetchall()
    times.append(time.perf_counter() - start)
print(f"after index: {statistics.mean(times) * 1000:.2f} ms")

The EXPLAIN QUERY PLAN output shows the access method clearly changing before and after the index is created.

before: SCAN users
after : SEARCH users USING INDEX idx_users_age (age=?)

Without an index, SQLite does a SCAN over the whole table; with the index, it switches to a SEARCH that goes directly to the matching rows. The measured query latency (average of 30 runs) was:

RowsWithout indexWith indexSpeedup
100,0003.95 ms0.87 ms~4.6x
1,000,00039.42 ms9.95 ms~4.0x

The speedup stays around 4x even as row count grows 10x. That’s because the age column here is low-cardinality (only 63 distinct values from 18 to 80), so even a single query still matches thousands of rows. (For a higher-cardinality column, like an email address, the index would provide an even larger speedup.) Still, the difference between a full scan and an index seek is clearly measurable here, confirming that CREATE INDEX isn’t just syntax — it genuinely changes the access path and speeds up the query.
